🥗 Nutri-Tsek

Full-stack nutrition-evaluation web and Android application

Technologies: Next.js, React, TypeScript, Tailwind CSS, ASP.NET Core, PostgreSQL, Supabase, Capacitor, Azure, and Vercel

Open Live Demo GitHub Repository Android Release

The web version demonstrates Nutri-Tsek’s main features. Barcode scanning and other mobile functions may work differently depending on the browser and device.

How the idea started

The idea for Nutri-Tsek came from two experiences.

While preparing for my Prøve i Dansk 3 (PD3) oral exam, my topic was “Sundhed i Danmark med fokus på nye madvaner.” Researching the topic made me think more about food choices, nutrition labels, and how difficult the information on food packaging can sometimes be to understand.

Another important experience happened during a summer trip to Switzerland. I was shopping with a friend when she scanned a drink using an application and showed me its product rating. I thought it was useful and interesting that an application could quickly explain information about a food product.

That experience stayed in my mind. Later, when I needed to choose a project for my Bachelor of Science in Computer Science thesis, I decided to explore the idea further.

What I wanted to build

I did not want Nutri-Tsek to show only one score. I wanted users to be able to examine food products from several perspectives.

The application includes:

  • Nutri-Score for overall nutritional quality
  • NOVA classification for the level of food processing
  • Nøglehullet assessment based on Nordic nutrition criteria
  • Traffic-light indicators for fat, saturated fat, sugars, and salt
  • Product comparison
  • Suggested alternatives based on available product data
  • Calorie and consumption tracking
  • Personal goals and recommendations

Users can scan a barcode or search for a product by name. Guest users can access basic product information, evaluation results, comparison features, and feedback submission. Registered users can access additional personalised features.

What made the project challenging

Building Nutri-Tsek was not only about writing code.

Food-product information can be incomplete, inconsistent, or different between data sources. I had to think carefully about how the application should respond when nutrition values, ingredients, or classification details were unavailable.

I also learned to be careful with the language used in the system. The application should not tell users that a product is always “good,” “bad,” or medically better for them. The results need to be presented neutrally, explained responsibly, and based only on the available product data.

The project required me to work with:

  • Frontend and backend development
  • REST APIs and databases
  • Authentication and password recovery
  • Mobile barcode scanning
  • Nutritional algorithms and rule-based processing
  • Cloud deployment
  • Testing and user feedback
  • Academic research and documentation
  • Application maintenance and version releases

What I learned

Nutri-Tsek helped me understand that building software involves much more than implementation.

I had to gather and define requirements, study nutrition-classification methods, design the system architecture and database, create the user interface, implement the business logic, test different cases, deploy the application, listen to feedback, and continue maintaining it.

The project also taught me that a technically working application can still need improvement. Users may misunderstand the wording, struggle with a feature, or expect something different from what I originally planned.

Because of that, I continued revising the system instead of treating the first working version as finished.

Nutri-Tsek is my largest project so far. It represents my Computer Science studies, my interest in solving practical problems, and my experience turning a personal idea into a complete web and mobile application.

A Danish speaking topic and a food-scanning experience in Switzerland inspired me to explore how software could make food information easier to understand.

📁 Folder Tree Viewer

Local developer tool for visualising project folder structures

Technologies: Python, Flask, HTML, JavaScript, Mermaid.js

GitHub Repository

Folder Tree Viewer runs locally because it reads folders directly from the user’s computer. A public live demo is therefore not required.

How the idea started

As my applications became larger, I sometimes found it difficult to keep track of all the folders and files in a project.

I wanted a quicker way to understand the complete project structure instead of opening each folder manually. That difficulty inspired me to build Folder Tree Viewer.

What I wanted to build

I wanted to create a simple local tool that could read a folder path and turn its structure into a visual Mermaid.js diagram.

The application includes:

  • Local folder-path scanning
  • Mermaid.js folder diagrams
  • Four layout directions
  • Zoom and reset controls
  • SVG download

The main challenge was reading nested folders and converting them into Mermaid.js syntax while keeping the final diagram understandable.

What I learned

This project taught me that a small difficulty in my own development workflow can become a useful software tool.

It also helped me understand how Python, Flask, HTML, JavaScript, and Mermaid.js can work together within one local application.

I built Folder Tree Viewer because I needed a clearer and faster way to understand the structure of my growing software projects.

🐍 Online Python Tutor — Modernised Fork

Modernised fork of an open-source Python code-visualisation tool

Technologies: Python, JavaScript, HTML, CSS

GitHub Repository

How the idea started

While learning programming, I sometimes found it difficult to understand what was happening inside a program as each line of code was executed.

Online Python Tutor helped by showing how variables, functions, and program execution change step by step. When I explored an existing Python 3 fork, I noticed that parts of the code were outdated and caused compatibility and security concerns on newer systems.

That encouraged me to study the existing codebase and modernise it.

What I wanted to do

This was not an application I created from the beginning. It is a fork based on an existing open-source project, so my goal was to improve it while respecting and acknowledging the work of the original authors and previous contributors.

My updates included:

  • Replacing the deprecated imp module with importlib
  • Fixing regular-expression escape warnings
  • Removing exposed API keys
  • Adding a homepage redirect to the visualiser
  • Verifying compatibility with Python versions 3.6 through 3.14
  • Retaining the original MIT licence and attribution

What made the project challenging

The main challenge was working with a large codebase that I did not originally create.

Before making changes, I needed to understand how the existing files and components worked together. I also needed to update the code carefully because changes to older components could affect existing behaviour.

Since this was an open-source fork, it was important to distinguish my contributions from the work of the original creator, previous maintainer, and other contributors.

What I learned

This project taught me that software development is not always about building a completely new application.

It can also involve reading unfamiliar code, maintaining an existing system, replacing deprecated components, improving security, and keeping software compatible with newer technology.

It also taught me to describe my contribution honestly and clearly distinguish my improvements from the work of the original creators.

I modernised an existing open-source code visualiser by updating deprecated code, improving security, and testing it with newer Python versions.

🥗 Nutri-Tsek

Full-stack web- og Android-applikation til vurdering af fødevarer

Teknologier: Next.js, React, TypeScript, Tailwind CSS, ASP.NET Core, PostgreSQL, Supabase, Capacitor, Azure og Vercel

Åbn live demo GitHub-repository Android-udgivelse

Webversionen demonstrerer Nutri-Tseks vigtigste funktioner. Stregkodescanning og andre mobilfunktioner kan fungere forskelligt afhængigt af browser og enhed.

Sådan opstod idéen

Idéen til Nutri-Tsek udsprang af to oplevelser.

Da jeg forberedte mig til den mundtlige del af Prøve i Dansk 3 (PD3), var mit emne “Sundhed i Danmark med fokus på nye madvaner.” Arbejdet med emnet fik mig til at tænke mere over fødevarevalg, ernæringsmærker og hvor svært det nogle gange kan være at forstå oplysningerne på fødevareemballage.

En anden vigtig oplevelse fandt sted på en sommerrejse til Schweiz. Jeg var ude at handle med en veninde, da hun scannede en drik med en app og viste mig produktets vurdering. Jeg syntes, det var både nyttigt og interessant, at en app hurtigt kunne forklare oplysninger om en fødevare.

Oplevelsen blev ved med at dukke op i mine tanker. Da jeg senere skulle vælge et emne til mit bachelorprojekt i datalogi, besluttede jeg at undersøge idéen nærmere.

Det ville jeg bygge

Jeg ønskede ikke, at Nutri-Tsek kun skulle vise én score. Jeg ville give brugerne mulighed for at undersøge fødevarer fra flere forskellige perspektiver.

Applikationen indeholder:

  • Nutri-Score til vurdering af den samlede ernæringsmæssige kvalitet
  • NOVA-klassifikation af graden af fødevareforarbejdning
  • Vurdering efter Nøglehullets nordiske ernæringskriterier
  • Trafiklysmarkering af fedt, mættet fedt, sukker og salt
  • Sammenligning af produkter
  • Forslag til alternativer baseret på tilgængelige produktdata
  • Registrering af kalorier og forbrug
  • Personlige mål og anbefalinger

Brugerne kan scanne en stregkode eller søge efter et produkt ved navn. Gæstebrugere har adgang til grundlæggende produktoplysninger, vurderingsresultater, sammenligningsfunktioner og mulighed for at indsende feedback. Registrerede brugere får adgang til yderligere personlige funktioner.

Det udfordrende ved projektet

Arbejdet med Nutri-Tsek handlede om meget mere end blot at skrive kode.

Oplysninger om fødevarer kan være ufuldstændige, inkonsistente eller variere mellem forskellige datakilder. Jeg måtte derfor nøje overveje, hvordan applikationen skulle reagere, når næringsværdier, ingredienser eller oplysninger til klassificering manglede.

Jeg lærte også at være omhyggelig med sproget i systemet. Applikationen bør ikke fortælle brugeren, at et produkt altid er “godt”, “dårligt” eller medicinsk set bedre. Resultaterne skal præsenteres neutralt, forklares ansvarligt og udelukkende baseres på de tilgængelige produktdata.

Projektet krævede, at jeg arbejdede med:

  • Frontend- og backendudvikling
  • REST API'er og databaser
  • Autentificering og nulstilling af adgangskoder
  • Stregkodescanning på mobilenheder
  • Ernæringsalgoritmer og regelbaseret behandling
  • Clouddeployment
  • Test og brugerfeedback
  • Akademisk research og dokumentation
  • Vedligeholdelse og versionsudgivelser

Det lærte jeg

Nutri-Tsek hjalp mig med at forstå, at softwareudvikling omfatter meget mere end selve implementeringen.

Jeg skulle indsamle og definere krav, undersøge metoder til ernæringsklassifikation, designe systemarkitekturen og databasen, udvikle brugergrænsefladen, implementere forretningslogikken, teste forskellige scenarier, deploye applikationen, lytte til feedback og fortsætte vedligeholdelsen.

Projektet lærte mig også, at en applikation kan fungere teknisk og stadig have brug for forbedringer. Brugerne kan misforstå en formulering, have svært ved en funktion eller forvente noget andet end det, jeg oprindeligt havde planlagt.

Derfor fortsatte jeg med at revidere systemet i stedet for at betragte den første fungerende version som færdig.

Nutri-Tsek er mit største projekt indtil videre. Det repræsenterer mit datalogistudie, min interesse for at løse praktiske problemer og min erfaring med at omsætte en personlig idé til en komplet web- og mobilapplikation.

Jeg lærte også at beskrive mit bidrag ærligt og tydeligt skelne mine forbedringer fra de oprindelige udvikleres arbejde.

Jeg moderniserede et eksisterende open source-værktøj til kodevisualisering ved at opdatere forældet kode, forbedre sikkerheden og teste det med nyere Python-versioner.
